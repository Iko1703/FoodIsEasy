1.3 Существующие программные средства разработки

Почему выбор программных средств так важен

На этапе проектирования сервиса персонализированного подбора блюд, планирования и анализа питания ключевым условием является выбор подходящих программных средств разработки. От корректно подобранного стека зависят скорость реализации функционала, качество кода, удобство сопровождения, безопасность обработки данных пользователей, а также возможность масштабирования системы в будущем. В условиях, когда сервис реализует как транзакционные операции (регистрация/авторизация, управление профилем, ведение истории), так и аналитические задачи (вычисление показателей рациона, формирование рекомендаций, агрегирование статистики), особенно важно обеспечить надежную платформу и средства для работы с реляционной структурой данных.

В данном дипломном проекте выбор программных средств основан на принципе соответствия технологических решений требованиям предметной области: наличие многомодульной бизнес‑логики, необходимость построения REST‑взаимодействия между клиентом и сервером, хранение связанных сущностей в базе данных и последующее развитие системы (в том числе добавление модулей рекомендаций и аналитики).

1.3.1 Средства разработки серверной части

Для серверной реализации веб‑сервиса используются технологии, обеспечивающие обработку запросов клиентов, управление бизнес‑логикой и доступ к базе данных. В практике разработки информационных систем наиболее распространены следующие подходы:

1) использование Java и JVM‑экосистемы;
2) использование JavaScript/TypeScript на сервере (Node.js);
3) использование Python‑платформ (Django, Flask, FastAPI и др.).

Сравнение и выбор конкретных инструментов

Java

В качестве основной серверной технологии выбран язык программирования Java. Выбор Java обусловлен следующими причинами:

— поддержка многопоточности и параллельной обработки запросов, что повышает устойчивость сервиса при росте числа пользователей;
— масштабируемость серверных приложений: JVM оптимизирована под работу с длительными процессами, а серверные контейнеры позволяют быстро разворачивать приложения в разных средах;
— зрелая экосистема библиотек, включающая средства для работы с REST‑API, безопасностью, сериализацией данных, интеграцией с БД и др.;
— развитые инструменты тестирования и профилирования (отладка, анализ производительности), которые помогают выявлять узкие места до выхода системы в эксплуатацию;
— ориентированность на корпоративную разработку, где важны стабильность, предсказуемость и поддерживаемость.

Spring Boot

Для построения серверного приложения используется фреймворк Spring Boot. Он существенно упрощает разработку благодаря:

— автоконфигурации типовых компонентов (веб‑слой, подключение к БД, настройки безопасности и т.д.);
— удобной организации структуры приложения (контроллеры, сервисы, репозитории/DAO);
— встроенной поддержке REST‑контроллеров, фильтров, обработчиков запросов и централизованного управления конфигурациями;
— интеграции с модулем Spring Security, что позволяет реализовать надежную аутентификацию и авторизацию.

Hibernate / JPA

Взаимодействие с базой данных реализовано через ORM‑подход, а именно с использованием спецификации JPA и реализации Hibernate. Данный подход позволяет:

— описывать доменную модель в виде сущностей Java и связывать их с таблицами БД;
— автоматически формировать SQL‑запросы и управлять загрузкой связанных данных;
— обеспечивать единообразный механизм работы с транзакциями и целостностью;
— уменьшать количество ручного SQL‑кода и снижать вероятность ошибок при изменении структуры данных.

REST API (JSON)

Сервис предоставляет клиенту интерфейс в формате REST API с использованием JSON. REST‑подход облегчает тестирование и развитие приложения, так как каждая операция соответствует HTTP‑методам (GET, POST, PUT/PATCH, DELETE) и отдельным URL‑ресурсам. Формат JSON является компактным и удобным для сериализации между сервером на Java и клиентом на JavaScript.

Почему Java подходит проекту FoodIsEasy

Сервис FoodIsEasy включает операции, требующие корректного управления данными и связями: хранение профилей пользователей, предпочтений, истории питания, планов по дням, связей пользователей с группами, голосований в группах и формирование корзины покупок. Java + Spring Boot позволяют надежно реализовать подобную бизнес‑логику, а Hibernate/JPA обеспечивает удобную работу с реляционной схемой.

При сравнении с Node.js и Python‑подходами можно отметить, что:

— Node.js часто выбирают для I/O‑ориентированных сервисов и быстрых прототипов, но в проекте требуется глубокая доменная логика и развитая инфраструктура безопасности/транзакций;
— Python удобен для аналитических вычислений и машинного обучения, однако для “ядра” сервиса с высокой нагрузкой на транзакционную работу и длительную поддержку архитектуры чаще выбирают Java‑подход.

Таким образом, Java‑платформа является обоснованным выбором для серверной части дипломного проекта.

1.3.2 Средства разработки базы данных (СУБД)

Для хранения информации о пользователях, их предпочтениях, рационе, блюдах, группах, голосованиях и корзинах покупок необходима надежная СУБД. В проекте выбрана PostgreSQL.

PostgreSQL

PostgreSQL является объектно‑реляционной СУБД, поддерживающей:

— транзакции ACID и гарантии целостности данных;
— сложные SQL‑запросы, необходимые для формирования статистики и расчетов показателей;
— внешние ключи и каскадные ограничения, что позволяет поддерживать согласованность связанных сущностей;
— расширяемость (функции, индексы, типы данных), что важно при дальнейшем развитии сервиса.

Сравнение PostgreSQL с MySQL / MongoDB

MySQL и PostgreSQL обе подходят для большинства веб‑проектов, однако PostgreSQL предоставляет более богатые возможности SQL и гибкость аналитических запросов. Это особенно актуально для сервиса, где требуется получение агрегированных данных (например, за период, по категориям, по разнообразию рациона).

MongoDB как NoSQL‑СУБД хорошо подходит для хранения документов и гибких схем. Однако доменная модель FoodIsEasy имеет выраженную реляционную структуру: связи “пользователь—предпочтения”, “план—позиции”, “группа—участники”, “голосование—варианты—участники”. Для таких сущностей реляционный подход повышает удобство контроля целостности и упрощает формирование запросов с использованием JOIN и агрегатов.

Выбор PostgreSQL обеспечивает устойчивость хранения данных и удобство дальнейшей аналитической обработки.

1.3.3 Средства разработки клиентской части

Клиентская часть реализуется как SPA (Single Page Application) — одностраничное приложение, в котором переходы между разделами выполняются без полной перезагрузки страницы. Для реализации UI выбран React.

React

React применяется для построения интерфейса на основе компонентов и декларативной логики отображения. В рамках проекта React предоставляет следующие преимущества:

— компонентный подход, который упрощает сопровождение сложного интерфейса (лента рекомендаций, таблицы/карточки блюд, формы профиля, экран планировщика питания);
— эффективное обновление интерфейса при изменении данных без лишних перезагрузок;
— удобство реализации маршрутизации и защищенных разделов (в интерфейсе предусмотрены отдельные страницы рекомендаций, питания, аналитики, групп и профиля);
— возможность интеграции с библиотеками управления состоянием (в проекте используется простое глобальное хранилище для токена авторизации).

Почему SPA подходит сервису

SPA‑модель удобна для динамических сценариев: рекомендации показываются лентой, план питания формируется по датам, элементы корзины изменяются в ответ на действия пользователя. Благодаря этому пользователь получает “плавный” опыт взаимодействия с сервисом и может быстрее выполнять задачи в рамках одного сеанса.

1.3.4 Средства обмена данными и формат взаимодействия

Ключевым каналом взаимодействия клиента и сервера является REST API. Данные передаются в формате JSON.

Использование REST API обеспечивает:

— прозрачность структуры интерфейса (операции над ресурсами);
— удобство тестирования с помощью запросов к URL‑эндпоинтам;
— простую интеграцию с браузером и различными клиентами (в дальнейшем возможны мобильные клиенты или интеграции).

Формат JSON выбран как стандарт обмена, так как:

— он хорошо поддерживается в экосистеме JavaScript;
— легко сериализуется и десериализуется на стороне Java;
— удобен для представления сложных структур (например, списков блюд, объектов планов, наборов параметров).

Для защиты доступа используется токенная аутентификация (JWT), что позволяет безопасно передавать сведения об авторизации между клиентом и сервером.

1.3.5 Инструменты разработки и сопутствующая инфраструктура

Помимо технологий разработки, важную роль играют инструменты, поддерживающие полный цикл разработки и эксплуатации.

Git

Система контроля версий Git применяется для:

— ведения истории изменений;
— работы по веткам и фиксации логически завершенных этапов;
— упрощения командной разработки (если она потребуется в будущем).

Maven

Для сборки и управления зависимостями проекта используется Maven. Он обеспечивает:

— централизованное описание зависимостей (pom.xml);
— воспроизводимость сборки;
— удобные команды для запуска приложения и тестов;
— упаковку приложения в артефакт (JAR), используемый для контейнерного развертывания.

IntelliJ IDEA

Интегрированная среда разработки IntelliJ IDEA предоставляет средства для:

— автодополнения и статического анализа;
— навигации по проекту и рефакторинга;
— отладки Java‑кода;
— удобной работы с проектом Maven и связанными библиотеками.

Postman (или аналогичные инструменты)

Для проверки REST‑эндпоинтов используются HTTP‑клиенты (например, Postman). Это позволяет:

— быстро тестировать API без фронтенда;
— анализировать коды ответов и корректность структуры JSON;
— документировать примеры запросов и сценарии.

Docker и Docker Compose

Для облегчения развертывания используется Docker и Docker Compose. Подход “всё в контейнерах” обеспечивает одинаковую среду выполнения на разных компьютерах. В рамках проекта контейнеризируются:

— PostgreSQL (база данных);
— backend (сервер приложения);
— frontend (клиентская часть).

Docker Compose связывает контейнеры в единый комплекс, задает параметры сети и порядок запуска, что снижает число проблем при установке и демонстрации проекта.

1.3.6 Итоговое обоснование выбора стека

Таким образом, выбранный стек программных средств — Java + Spring Boot (REST API и безопасность), Hibernate/JPA (ORM‑доступ к данным), PostgreSQL (реляционная СУБД), React (SPA‑интерфейс рекомендаций, планирования и аналитики), а также инструменты Git, Maven, IntelliJ IDEA, Postman и Docker — полностью соответствует задачам сервиса FoodIsEasy.

Сочетание этих технологий обеспечивает:

— надежную серверную архитектуру и удобную поддержку бизнес‑логики;
— корректную работу с реляционной моделью данных и возможность аналитических вычислений;
— современный пользовательский интерфейс с динамической подачей информации;
— практичность разработки, тестирования и развертывания в единой среде.

Следовательно, выбранные программные средства являются обоснованными и обеспечивают основу для дальнейшего развития сервиса, включая расширение модулей аналитики и механизмов персонализированных рекомендаций (в том числе с привлечением методов машинного обучения).
